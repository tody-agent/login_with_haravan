## 2025-05-01 - Missing Authorization Checks on API Endpoints
**Vulnerability:** Setup and configuration functions in `login_with_haravan/setup/install.py` were exposed to the public API using `@frappe.whitelist()`, but lacked role-based access control.
**Learning:** In the Frappe framework, `@frappe.whitelist()` exposes functions to the API, allowing any authenticated user to potentially trigger them. It does not enforce role checks automatically.
**Prevention:** Always add explicit authorization checks, such as `frappe.only_for("System Manager")` or `frappe.has_permission(...)`, inside whitelisted functions that perform sensitive operations like modifying system configuration or metadata.

## 2026-05-17 - Improper Patching of get_permission_query_conditions Leading to IDOR
**Vulnerability:** A monkey-patch for Frappe's `user.get_permission_query_conditions` in `login_with_haravan/hooks.py` bypassed authorization by always returning an empty string, intending to fix an SQL Ambiguous Column error in cross-table joins but inadvertently causing an IDOR vulnerability by granting full read access to all users' data to anyone.
**Learning:** Returning an empty string in a Frappe permission condition overrides all row-level access restrictions for that doctype. If patching permission queries to fix SQL issues (like "ambiguous column 'name'"), one must retain authorization restrictions by verifying privileges using `frappe.has_permission()` or explicitly adding user checks with proper table alias prefixing like ```tabDoctype`.name``` to prevent SQL syntax errors in joins.
**Prevention:** Never return a broad bypass (like `return ""`) for all users in `get_permission_query_conditions`. Verify read permissions dynamically via `frappe.has_permission(doctype, ptype='read', user=user_name)` to allow privileged users, and for standard users, return properly prefixed SQL checks (e.g. ``return f"`tabUser`.name = {frappe.db.escape(user_name)}"``) that maintain row-level security while preventing ambiguous column errors.
