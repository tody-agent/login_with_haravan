# Proposal: Ticket CC Emails
Generated by: cm-brainstorm-idea
Date: 2026-04-30

## Why we're doing this

### Qualified Problem

**For:** Haravan support agents who create or reply to Helpdesk tickets on behalf of customers.
**Who:** Need to copy store owners, stakeholders, or internal customer contacts into ticket email threads.
**The:** Current Haravan Helpdesk ticket flow stores the requester and contact, but does not have a durable ticket-level CC list.
**That:** Should behave like the previous Freshdesk experience: when an agent creates a customer ticket, optional CC recipients receive ticket email updates and continue to receive agent replies.
**Unlike:** A one-off email sent at creation time or a UI-only CC value, the CC list must be persisted on `HD Ticket` and applied to outgoing ticket emails.
**Our approach:** Add `custom_cc_emails` to `HD Ticket`, expose it to agent ticket creation, normalize and validate it server-side, and merge it into Helpdesk's outgoing ticket reply workflow from source-controlled app code.

### Root Causes

1. Technical: `HD Ticket` has no source-controlled custom field for ticket-level CC recipients.
2. Product: Agent-created ticket flow does not collect CC, even though support workflows often involve multiple customer-side stakeholders.
3. Design: The existing agent ticket dialog has customer email, phone, customer, source channel, type, priority, subject, and description, but no lightweight CC affordance.
4. Operational: Several Helpdesk automations are Server Scripts; relying only on another Server Script for CC would be hard to test and easy to drift from Git.

### Impact If Not Addressed

- Agents must manually forward or repeat updates to additional contacts.
- Customer stakeholders miss context, causing duplicate tickets or slower resolution.
- Haravan Helpdesk remains less familiar than the previous Freshdesk workflow.
- Email behavior becomes inconsistent if agents add CC manually in one place but later replies omit it.

## Current System Checks

### Codebase Summary

- Tech stack: Frappe custom app `login_with_haravan`, Python backend, Frappe hooks, Helpdesk Vue SPA assets, Server Script / Client Script operations on production.
- Architecture: follows local 7-layer style: DocType metadata in app, business logic under `engines/`, whitelisted APIs in Python modules, install/migrate setup in `setup/install.py`, tests under `login_with_haravan/tests/`.
- Current hooks: `after_install`, `after_migrate`, and `HD Ticket.before_insert` for `auto_set_customer`.
- Current agent flow: production has `Onboarding - Agent Ticket Dialog` and `Onboarding - Agent Ticket API`/`haravan_agent_create_customer_ticket` script inventory; the dialog currently collects customer email, phone, customer, store URL, channel, send email, type, priority, subject, and description.
- Test gate: `npm run test` runs `./test_gate.sh`, repo secret checks, and Python unittest discovery.
- Corpus size: about 38 source files and 138 docs/OpenSpec files, excluding `docs/node_modules`.

### haravan.help Checks

- `https://haravan.help/helpdesk/my-tickets/new` returns the Helpdesk Vue SPA publicly and routes unauthenticated users to login.
- The SPA creates tickets through `helpdesk.helpdesk.doctype.hd_ticket.api.new`, which inserts an `HD Ticket` with dynamic template fields.
- `HD Ticket Template` REST access is correctly blocked for Guest, so field/template verification requires an authenticated Desk session.
- Public Helpdesk assets show ticket creation fields are pulled from `HD Ticket Template > Default` via `helpdesk.helpdesk.doctype.hd_ticket_template.api.get_one`.
- Local Helpdesk source confirms `HDTicket.reply_via_agent()` creates a `Communication`, then calls `frappe.sendmail(... cc=cc ...)` using the method argument.

### Assessment Of Suggested Manual Plan

- Step 1, Custom Field: Valid, but this should be idempotently managed by the app instead of only manual Desk setup.
- Step 2, HD Ticket Template: Valid. For this request, default should be agent-visible and customer-hidden unless product explicitly wants customers to enter CC too.
- Step 3, Send CC on create: Partially valid. A separate creation email can notify CC contacts, but it is not the same as being CC'd on the native acknowledgement email and can duplicate notifications.
- Step 4, Communication Before Save: Risky/incomplete. In current Helpdesk, modifying `Communication.cc` in a DocEvent does not guarantee the outgoing email includes that CC because `reply_via_agent()` passes its local `cc` argument into `frappe.sendmail`.
- Step 5, HD Form Script manage CC: Useful later, but not enough for agent-created ticket parity. Editing CC after creation should be supported once the email pipeline is correct.

## Options Evaluated

### Option A: Manual Desk + Server Scripts

**Approach:** Create `custom_cc_emails`, add it to `HD Ticket Template`, then add Server Scripts for create notification and `Communication` CC merge.
**Philosophy:** Quick operational patch.

**What changes:**
- Desk config: Custom Field and template row.
- Server Script: create CC notification and attempt to merge CC on Communications.

**Effort:** S (~0.5-1 day)
**Risk level:** High

**Pros:**
- Fastest to try on production.
- No app deploy required if Server Script is enabled.

**Cons:**
- The suggested `Communication / Before Save` hook is not sufficient for `reply_via_agent()` email delivery.
- Harder to test locally and easy to drift from Git.
- Can duplicate create notifications if acknowledgement email is also customized later.

**Best when:** A temporary smoke-test is needed before code work, and operators accept manual rollback.

### Option B: Source-Controlled App Extension + Minimal Desk Config

**Approach:** Add the custom field/template row idempotently in `setup/install.py`, implement CC parsing/validation/merge logic in app code, and extend `HD Ticket` email behavior through a source-controlled class override or equivalent wrapper.
**Philosophy:** Balanced and production-safe.

**What changes:**
- `setup/install.py`: create `HD Ticket-custom_cc_emails`; optionally add it to `HD Ticket Template > Default`.
- `engines/ticket_cc.py`: parse comma/semicolon/newline input, validate emails, normalize case, de-duplicate, and remove primary recipients.
- `hooks.py`: add `HD Ticket` validation / after-insert hook and a safe `HD Ticket` controller extension or override.
- `overrides/hd_ticket.py`: merge ticket CC into `reply_via_agent()` before calling Helpdesk's native sendmail path; include CC in agent-created acknowledgement when applicable.
- Production scripts: update `Onboarding - Agent Ticket Dialog` and `haravan_agent_create_customer_ticket` to accept/pass `custom_cc_emails`, or migrate that API into app code.
- Tests: pure unit tests for normalization/merge plus mocked tests for reply/create behavior.

**Effort:** M (~1.5-2.5 days)
**Risk level:** Medium

**Pros:**
- Email behavior is correct because CC is passed before `frappe.sendmail()` builds `Email Queue`.
- Reuses Helpdesk's native reply threading, templates, sender, delayed/instant send settings, and communication timeline.
- Deployable and testable through existing `npm run test`.
- Avoids a pile-up of fragile Server Scripts.

**Cons:**
- Requires app deploy/migrate.
- `override_doctype_class` must be checked for conflicts with other apps before implementation.
- Template row creation must be careful because production form customization already has many custom fields.

**Best when:** This feature should become a stable Freshdesk-parity capability.

### Option C: First-Class CC Child DocType

**Approach:** Create a child table like `HD Ticket CC Recipient` instead of a comma-separated field, with per-recipient status, source, and audit trail.
**Philosophy:** Long-term product model.

**What changes:**
- New child DocType and ticket table field.
- Richer UI for add/remove CC recipients.
- More complex email merge and possible inbound-reply handling.

**Effort:** L (~4-6 days)
**Risk level:** Medium-High

**Pros:**
- Best data model for many recipients, audit, future unsubscribe, and permissions.
- Can evolve toward Freshdesk-like participant management.

**Cons:**
- Overbuilt for the current request.
- Harder to expose through current `HD Ticket Template` dynamic fields.
- More moving parts in Desk, Helpdesk SPA, and API scripts.

**Best when:** Haravan wants full participant management, not just ticket-level CC parity.

## Recommendation

**Choose Option B: Source-Controlled App Extension + Minimal Desk Config.**

### Why Option B

1. It solves the real email-delivery problem. Current Helpdesk passes the local `cc` argument into `frappe.sendmail`, so the app must merge ticket CC before that call.
2. It fits the repo's architecture: metadata in setup, pure parsing in engines, hooks/overrides in app code, tests in `login_with_haravan/tests`.
3. It gives Freshdesk-like continuity without introducing a full participant subsystem.
4. It keeps production operations clean: Desk config remains minimal, while critical behavior is versioned and testable.

### Why Not The Others

- **Option A:** Good for a quick manual experiment, but the suggested Communication hook can miss actual email CC delivery.
- **Option C:** More future-proof, but too heavy for this immediate parity need.

### Quick Win

Start with Option B but keep the UI simple:

- Field label: `CC Emails`
- Fieldname: `custom_cc_emails`
- Field type: `Small Text`
- Placeholder: `email1@company.com, email2@company.com`
- Default template visibility: `hide_from_customer = 1` for launch, so agents can use it while customer portal remains simpler.

## Acceptance Criteria

- Agent can create a customer ticket with optional `CC Emails`.
- `HD Ticket.custom_cc_emails` persists normalized valid email addresses.
- Invalid CC input is rejected with a clear message before ticket creation or save.
- Agent reply email includes ticket CC recipients in the actual email headers.
- CC recipients are not duplicated if already in To/CC.
- Customer portal does not expose CC by default unless explicitly enabled.
- Existing ticket creation, AI/GitLab scripts, customer profile panel, and Haravan login continue to pass the test gate.
