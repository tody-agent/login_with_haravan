# Proposal: Bitrix Metajson Customer Enrichment
Generated by: cm-brainstorm-idea
Date: 2026-05-05

## Why we're doing this

### Qualified Problem

**For:** Haravan Helpdesk agents and support automation that read customer context through metajson.

**Who:** need complete and current company data, but today Bitrix enrichment is mostly lazy/on-demand through the customer profile panel.

**The:** Login With Haravan custom app is a Frappe integration layer.

**That:** should enrich `HD Customer` from Bitrix by Haravan `orgid` whenever metajson is queried, without making login, ticket load, or repeated reads expensive.

**Unlike:** a blind write-on-every-request sync, the workflow should be idempotent, TTL-guarded, diff-aware, and silent when there is no `orgid`.

**Our approach:** trigger a bounded Bitrix company lookup from the metajson path, then upsert `HD Customer` and `HD Customer Data` only when Bitrix has newer or different data.

### Root Causes

1. Bitrix company data is useful to Helpdesk, but current app stores it mainly when an agent opens the customer profile enrichment flow.
2. `HD Customer` has core Bitrix fields (`custom_bitrix_company_id`, URL, status, confidence, last synced), but not enough sync guard metadata to distinguish "checked recently and unchanged" from "needs refresh".
3. A metajson trigger can be high-frequency, so a naive implementation would create loops, extra writes, and avoidable Bitrix API cost.

### Impact if NOT addressed

- Agents and automations may see incomplete customer profiles until someone manually opens or refreshes a profile.
- Frappe Helpdesk customer records can become stale after Bitrix company changes.
- Later AI or routing logic using metajson may make poorer decisions because customer context is thin.

## What's changing

### Option A: Synchronous Smart Enrichment in Metajson

**Approach:** When metajson is queried, extract `orgid`, find the linked `HD Customer`, and run a Bitrix company lookup only if TTL/diff rules say it is worth doing. Save only changed fields and snapshot payloads.

**Philosophy:** Balanced.

**What changes:**
- Add a reusable engine method such as `sync_bitrix_company_for_orgid(orgid, source="metajson")`.
- Use `bitrix_refresh_ttl_minutes` as the minimum interval for non-forced checks.
- Add guard fields on `HD Customer`:
  - `custom_bitrix_last_checked_at`
  - `custom_bitrix_company_modified_at`
  - `custom_bitrix_payload_hash`
  - optionally `custom_bitrix_not_found_at`
- Reuse `HD Customer Data` as the normalized snapshot store.
- If no `orgid`, return without logging.
- If duplicate Bitrix companies are returned, accept the first result from `DATE_MODIFY DESC`.
- If Bitrix payload hash and `DATE_MODIFY` are unchanged, skip Frappe save.

**Effort:** M (~0.5-1 day)

**Risk level:** Low/Medium

**Pros:**
- Fresh data appears exactly where metajson consumers need it.
- Simple operational model, no worker dependency.
- Easy to test with mocks.

**Cons:**
- Metajson request may wait for Bitrix when TTL expires.
- Needs strict timeout and fail-soft behavior.

**Best when:** metajson is not called at extremely high volume and freshness during the same request matters.

### Option B: Metajson Enqueues Background Sync

**Approach:** Metajson extracts `orgid`, returns immediately using existing Helpdesk data, and enqueues a background job to refresh Bitrix data if TTL expired.

**Philosophy:** Tech-first.

**What changes:**
- Add a queue-safe sync method and job de-dup key by `orgid`.
- Use cache lock such as `bitrix:company-sync:{orgid}` to prevent parallel jobs.
- Metajson response can include stale current data while job updates later.

**Effort:** M/L (~1-1.5 days)

**Risk level:** Medium

**Pros:**
- Keeps metajson fast.
- Handles bursty traffic better.

**Cons:**
- Data is eventually consistent, not fresh in the current response.
- More moving parts: queue, lock, job visibility, retry behavior.

**Best when:** metajson is performance-critical or called at high volume.

### Option C: Scheduled Reconciliation Plus Lightweight Metajson

**Approach:** Run scheduled jobs to refresh known `HD Customer` records, while metajson only reads local data and optionally triggers sync for never-seen orgs.

**Philosophy:** Business-first operations.

**What changes:**
- Add a scheduler job scanning customers with `custom_haravan_orgid`.
- Refresh stale records in batches with rate limits.
- Metajson path avoids network calls except for first-time customers.

**Effort:** L (~1.5-2 days)

**Risk level:** Medium/High

**Pros:**
- Predictable API usage and broad coverage.
- Good long-term data hygiene.

**Cons:**
- More operational complexity and slower first update.
- Risk of scanning too many customers unless batching is carefully tuned.

**Best when:** the system has many orgs and wants proactive hygiene beyond metajson usage.

## Recommendation

**Choose Option A: Synchronous Smart Enrichment in Metajson**, with one escape hatch: if metajson latency becomes an issue, the same engine method can later be moved behind Option B's background queue.

### Why Option A

1. It directly satisfies "mỗi khi trigger truy vấn metajson thì trigger gọi api bitrix companies theo orgid" while still avoiding an API call every time through TTL.
2. The app already has the right architecture: `BitrixClient`, `customer_enrichment`, `HD Customer Data`, config TTL, and setup-managed custom fields.
3. It is the smallest change that keeps data current and avoids update loops, because it does not hook into `HD Customer.save()` or login callbacks.

### Why not the other options first

- **Option B:** Good fallback for scale, but it may not enrich data in the same metajson response.
- **Option C:** Useful later for data hygiene, but too broad for the requested trigger and can consume more Bitrix quota.

## Proposed Smart Flow

1. Metajson endpoint receives/query-builds context.
2. Extract `orgid` from request, ticket, account link, or existing `HD Customer.custom_haravan_orgid`.
3. If `orgid` is missing, return normally with no log.
4. Resolve `HD Customer` by `custom_haravan_orgid`; if missing and Bitrix returns a company, create or upsert `HD Customer` using deterministic Haravan naming if enough fields exist, otherwise store only safe Bitrix references.
5. Check `custom_bitrix_last_checked_at` against `bitrix_refresh_ttl_minutes`.
6. If TTL has not expired and caller did not force refresh, skip Bitrix call.
7. Call Bitrix `crm.company.list` filter by Haravan org id, select `*`, `UF_*`, `EMAIL`, `PHONE`, `WEB`, order `DATE_MODIFY DESC`.
8. If no result, set `custom_bitrix_last_checked_at` and optionally `custom_bitrix_not_found_at`; do not log.
9. If multiple results, take first result because Bitrix query is ordered newest first.
10. Compute stable payload hash from normalized company payload.
11. Compare:
    - Bitrix `DATE_MODIFY` versus `custom_bitrix_company_modified_at`.
    - payload hash versus `custom_bitrix_payload_hash`.
    - mapped HD Customer fields versus current values.
12. If no meaningful change, update only `custom_bitrix_last_checked_at` if needed and skip broader save.
13. If changed, update `HD Customer` fields and upsert `HD Customer Data` snapshot.
14. Return metajson with local enriched fields.

## Loop and Resource Safety Rules

- Do not trigger from `HD Customer.on_update`, `HD Customer Data.on_update`, or login callback.
- Use metajson as the only initial trigger for this workflow.
- Use TTL guard before network calls.
- Use payload hash and Bitrix `DATE_MODIFY` before writes.
- Use a short cache lock keyed by org id if parallel metajson calls are possible.
- Use existing Bitrix timeout config and fail soft on exceptions.
- Log only unexpected API/runtime errors; do not log missing org id or no Bitrix match.
- Do not write if values are identical.

## Suggested Field Additions

Add to `HELPDESK_PROFILE_CUSTOM_FIELDS["HD Customer"]`:

| Fieldname | Type | Purpose |
| --- | --- | --- |
| `custom_bitrix_last_checked_at` | Datetime | Last time metajson considered/refreshed this org, including no-change checks. |
| `custom_bitrix_company_modified_at` | Datetime | Latest Bitrix `DATE_MODIFY` seen for the matched company. |
| `custom_bitrix_payload_hash` | Data | Stable hash of normalized company payload to skip identical saves. |
| `custom_bitrix_not_found_at` | Datetime | Optional. Last time Bitrix returned no company for the org id. |
