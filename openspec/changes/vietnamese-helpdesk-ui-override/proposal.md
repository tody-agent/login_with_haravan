# Proposal: Vietnamese Helpdesk UI Override
Generated by: cm-brainstorm-idea
Date: 2026-05-01

## Why we're doing this

### Qualified Problem

**For:** Haravan customers using the Helpdesk portal in Vietnamese.
**Who:** Need to search, filter, create, and track support tickets without mixed English/Vietnamese UI labels.
**The:** Frappe Helpdesk portal currently renders several list and ticket-creation controls in English even when the user chooses Vietnamese.
**That:** Should display Vietnamese labels for common ticket-list, search, filter, sort, column, status, priority, and button text on the customer-facing portal.
**Unlike:** Native Frappe translations or field-label configuration, which do not fully cover the Helpdesk Vue SPA strings visible in `/helpdesk/my-tickets` and `/helpdesk/my-tickets/new`.
**Our approach:** Add a small, isolated, source-controlled frontend override in the custom app that activates only for Vietnamese users and only on the Helpdesk customer routes.

### Root Causes

1. Technical: Some Helpdesk portal labels are rendered by the Vue SPA or third-party components and are not reliably covered by standard Frappe translation records.
2. Product: Customer-facing support workflows look unfinished when core actions like `Create`, `Filter`, `Columns`, `Search`, `Status`, and `Priority` remain in English.
3. Design: Mixed-language controls add friction exactly where users are scanning lists, filtering tickets, or submitting a new support request.
4. Operational: Manual Desk translations or ad hoc scripts are hard to audit and hard to remove later.

### Impact If Not Addressed

- Vietnamese customers see a partially localized portal, lowering trust in the support experience.
- Ticket discovery becomes slower because search/filter/sort controls are not in the user's chosen language.
- Operators keep accumulating one-off fixes that are difficult to trace when Frappe Helpdesk eventually ships better Vietnamese support.

## Current System Checks

### Codebase Summary

- Tech stack: Frappe custom app `login_with_haravan`, Python backend, Frappe hooks, source-controlled `web_include_js`, and Helpdesk Vue SPA enhancement scripts.
- Architecture: Current app follows the local 7-layer style: DocType metadata under `login_with_haravan/login_with_haravan/doctype/`, business logic under `engines/`, whitelisted APIs in Python modules, setup in `setup/install.py`, tests under `login_with_haravan/tests/`, and client scripts under `public/js/`.
- Existing frontend injection: `hooks.py` includes `haravan_login_redirect.js`, `haravan_org_selector.js`, and `customer_profile_panel.js` through `web_include_js`.
- Existing guard pattern: `haravan_org_selector.js` already activates only on `/helpdesk/my-tickets/new` and reruns on SPA navigation.
- Test gate: `npm run test` runs `./test_gate.sh`, repo safety checks, and Python unittest discovery.
- Corpus size: about 42 source files and 183 docs/OpenSpec files excluding `docs/node_modules`; generated VitePress cache/dist files inflate docs count.

### UI Scope From Screenshots

Observed untranslated or partially untranslated labels on `/helpdesk/my-tickets`:

- Primary action: `Create`
- Toolbar: `Filter`, `Last Modified`, `Columns`
- Search/filter placeholders: `ID`, `Subject`, `Search`
- Table headers: `Subject`, `Status`, `Priority`, `First response`, `Resolution`, `Team`
- Ticket values: `Open`, `Resolved`, `Medium`
- Relative times: `in 4 days`, `in 11 hours`, `12 hours ago`, `a day ago`
- Filter menu: `Empty - Choose a field to filter by`, `Add Filter`, `Resolution By`, `Response By`

The override should start with these visible strings and then expand only through a controlled dictionary.

## Options Evaluated

### Option A: Native Translation Records Only

**Approach:** Add or update Frappe `Translation` records, fixtures, or `.po` files for Helpdesk labels and rely on the framework translation layer.
**Philosophy:** Platform-first.

**What changes:**

- Setup/fixtures: Create translation records for known Helpdesk strings.
- Deployment: Run migrate/build and clear cache.
- Operations: Continue adding translations as missing labels appear.

**Effort:** S-M (~0.5-1.5 days)
**Risk level:** Medium

**Pros:**

- Cleanest if the strings pass through Frappe's translation functions.
- Minimal frontend mutation.
- Easy to keep after upstream improves.

**Cons:**

- Does not reliably cover Vue-rendered component labels, placeholders, menu items, and third-party controls.
- Hard to verify every string path without patching Helpdesk source.
- User has already tried multiple approaches; this is unlikely to solve the visible gap alone.

**Best when:** The remaining labels are known to pass through Frappe's standard translation system.

### Option B: Isolated Frontend Vietnamese Override

**Approach:** Add one source-controlled JS module that activates only when the active language is Vietnamese and the route is `/helpdesk/my-tickets` or `/helpdesk/my-tickets/new`. It uses a small dictionary plus a `MutationObserver` to translate text nodes, input placeholders, button labels, dropdown menu items, table headers, and known relative-time fragments after the SPA renders.
**Philosophy:** Last-resort but reversible.

**What changes:**

- `public/js/helpdesk_vi_override.js`: New isolated module with `window.HaravanHelpdeskViOverride` namespace.
- `hooks.py`: Add a single `web_include_js` entry after existing Helpdesk portal scripts.
- Dictionary: Keep all forced strings in one object, grouped by `actions`, `toolbar`, `filters`, `columns`, `values`, and `relative_time`.
- Activation guards: Check route, Vietnamese locale, and an optional kill-switch flag before observing DOM.
- Tests: Add frontend safety checks for route guard, locale guard, mutation observer, dictionary presence, and no broad global replacement on non-Helpdesk pages.

**Effort:** M (~1-2 days)
**Risk level:** Medium

**Pros:**

- Directly fixes the customer-visible mixed-language problem.
- No Helpdesk core fork and no mutation of upstream source.
- Easy to remove later: delete one hook include and one JS file.
- Fits the repo's existing pattern for Helpdesk SPA enhancements.
- Can be verified visually against the two target routes.

**Cons:**

- DOM text replacement can be brittle if Helpdesk changes markup or labels.
- Must avoid translating user-generated ticket content such as subjects and descriptions.
- Needs careful scoping so English users and agent Desk pages are not affected.

**Best when:** Native translation has failed or is incomplete, but the business needs a polished Vietnamese customer portal now.

### Option C: Fork Or Patch Frappe Helpdesk Frontend

**Approach:** Patch Helpdesk's Vue source or maintain a fork with proper translation calls for every missing label, then build/deploy the patched Helpdesk app.
**Philosophy:** Correct at source.

**What changes:**

- Helpdesk frontend source: Add missing i18n wrappers and Vietnamese messages.
- Build/deploy pipeline: Maintain custom Helpdesk asset build.
- Upstream strategy: Periodically rebase or submit PRs upstream.

**Effort:** L-XL (~4-10 days plus ongoing maintenance)
**Risk level:** High

**Pros:**

- Most semantically correct implementation.
- Avoids DOM mutation.
- Could benefit upstream if contributed.

**Cons:**

- Violates the current project rule to avoid modifying Frappe/Helpdesk core for this integration.
- More deployment risk on Frappe Cloud.
- Harder to remove than a custom app override.
- Increases maintenance every time Helpdesk updates.

**Best when:** Haravan decides to own a Helpdesk fork or contribute localization upstream.

## Recommendation

**Choose Option B: Isolated Frontend Vietnamese Override.**

### Why Option B

1. It matches the stated constraint: this is the final fallback after native translation attempts, and it can be removed later without touching Helpdesk core.
2. It fits the current app architecture: a Layer 7 client JS module with route guards, no server business logic, and source-controlled tests.
3. It solves the exact screenshots: button names, search placeholders, filter menus, table headers, status/priority labels, and visible relative-time fragments.
4. It gives operations a clean rollback path: remove `/assets/login_with_haravan/js/helpdesk_vi_override.js` from `web_include_js`, build, clear cache.

### Why Not The Others

- **Option A:** Worth keeping as a long-term preference, but it is not reliable enough for the Vue SPA strings shown in the screenshots.
- **Option C:** Technically cleaner at source, but too costly and too sticky for a temporary localization bridge.

### Quick Win

Start with Option B for only:

- `/helpdesk/my-tickets`
- `/helpdesk/my-tickets/new`
- Vietnamese locale users only
- A first dictionary covering the screenshot strings

Do not attempt to translate arbitrary page text or user-generated ticket data in the first version.

## Proposed Implementation Shape

### Module Boundary

Create a removable module:

```text
login_with_haravan/public/js/helpdesk_vi_override.js
```

Register it last in:

```text
login_with_haravan/hooks.py
```

The module should own:

- Locale detection: `frappe.boot.lang`, `frappe.boot.user.language`, `document.documentElement.lang`, and browser fallback.
- Route detection: exact customer portal routes only.
- Dictionary: exact source string to target string mappings.
- Attribute translation: `placeholder`, `title`, `aria-label`, and visible text for safe controls.
- DOM observer: debounced `MutationObserver` for Helpdesk SPA rerenders and dropdowns.
- Safety filters: skip `input` values, textareas, rich text editors, ticket subject cells, description content, and any element marked with `data-haravan-no-translate`.

#### Relative Time Patterns

Relative-time translation should be pattern-based and conservative:

- `in N hours` -> `trong N giờ`
- `in N days` -> `trong N ngày`
- `N hours ago` -> `N giờ trước`
- `a day ago` -> `1 ngày trước`
- `N minutes ago` -> `N phút trước`
- `a minute ago` -> `1 phút trước`
- `N days ago` -> `N ngày trước`
- `N months ago` -> `N tháng trước`
- `N years ago` -> `N năm trước`
- `in N minutes` -> `trong N phút`
- `in N months` -> `trong N tháng`
- `in N years` -> `trong N năm`

#### Do Not Translate Automatically

The DOM override must skip these even if they contain English:

- Ticket subject, description, comments, rich text editor content, and attachment names.
- Knowledge base article titles, headings, excerpts, and body content because these are editorial/user content.
- Customer names, contact names, team names, organization names, domains, email addresses, phone numbers, and Haravan IDs.
- Technical tokens: `@me`, fieldnames, route names, CSS classes, API method names, doctype names, and hidden form values.
- Already Vietnamese custom fields such as `Nhóm dịch vụ`, `Đối tác cung cấp dịch vụ`, `Trạng thái thanh toán`, `Số điện thoại`, and product-specific labels configured in production.

### Removal Plan

When Frappe Helpdesk supports Vietnamese sufficiently:

1. Remove the script from `web_include_js`.
2. Delete `public/js/helpdesk_vi_override.js`.
3. Delete related tests.
4. Build/clear cache.
5. Keep this proposal as the historical reason for removal.
