### Key Architecture Insight

The screenshot from frappedesk.com (Frappe's own support portal) shows a **critical UX pattern**:

> When a user belongs to multiple organizations ("sites"), the ticket creation form displays a **"Site Name" dropdown** that lets the user **select which organization** they are filing the ticket for.

This is exactly the pattern we need to replicate for Haravan organizations.

### How Frappe Helpdesk Already Models This

From the HD Ticket DocType schema:

```
contact     → Link to "Contact" DocType
customer    → Link to "HD Customer" DocType
raised_by   → Email (Data field)
```

`HD Customer` has fields: `customer_name`, `domain`, `image`. It's a lightweight org/company entity.

**The pattern is clear:**
- **Contact** = the person (linked to User via email)
- **HD Customer** = the organization
- **HD Ticket** links to both
- Portal form shows a dropdown of `HD Customer` records the Contact belongs to

### Current Gaps

| Gap | Impact |
|-----|--------|
| `enrich_frappe_data()` creates **Company** docs, not **HD Customer** docs | Helpdesk won't see them in the ticket form |
| No API to list Haravan orgs for current user | Portal can't populate the org dropdown |
| `Haravan Account Link` stores mapping but isn't queryable from frontend | No way for portal JS to know which orgs belong to logged-in user |
| Debug file-dump code still in `oauth.py` (line 94-100) | Must be removed before production |
| No automatic Contact → HD Customer linkage | Tickets created via portal won't have correct customer |

### Qualified Problem

**For:** Haravan merchants managing multiple stores/organizations
**Who:** Need to file support tickets specifying which store the issue is for
**The:** `login_with_haravan` Frappe app is a social login bridge
**That:** Should automatically create/link HD Customer records and expose an org picker in the ticket form
**Unlike:** Current state where only Company docs are created (invisible to Helpdesk) and there's no multi-org support
**Our approach:** Use Helpdesk's native HD Customer + Contact model, augmented with a whitelisted API for listing linked orgs

### Root Causes

1. **Technical:** `enrich_frappe_data()` creates `Company` docs instead of `HD Customer` docs — wrong DocType for the Helpdesk module
2. **Product:** No mechanism for the user to "switch" between orgs or select which org a ticket is for
3. **Design:** The portal ticket creation form has no Haravan-org-aware dropdown

### Impact if NOT Addressed

- **Business:** Merchants file tickets without org context → agents waste time asking "which store?" → slower resolution
- **User:** Frustration — "I manage 4 stores but can only file tickets for the one I logged in with"
- **Technical:** Company docs accumulate without being used by Helpdesk, creating data fragmentation

## Phase 3: DEVELOP — 3 Solution Options

### Option A: HD Customer Native Integration (Recommended)

**Approach:** Replace Company-based enrichment with HD Customer. Add whitelisted API for org listing. Add custom field to HD Ticket for org selection. Use Helpdesk's native Contact→HD Customer linking.

**Philosophy:** Platform-native, minimal friction

**What changes:**

| Component | Change |
|-----------|--------|
| `oauth.py` → `enrich_frappe_data()` | Create/update `HD Customer` instead of `Company`. Link Contact to HD Customer. |
| `Haravan Account Link` DocType | Add `hd_customer` Link field to track which HD Customer was created for each org. |
| New engine: `engines/sync_helpdesk.py` | Dedicated sync logic: upsert HD Customer, link Contact, set default customer. |
| New API: `oauth.py` → `get_user_orgs()` | Whitelisted endpoint returning list of Haravan orgs (HD Customers) for logged-in user. |
| Hooks: `doc_events` on HD Ticket | Auto-populate `customer` field from user's default/selected org on ticket creation. |
| Frontend: `haravan_org_selector.js` | Inject org picker into Helpdesk portal ticket form (similar to "Site Name" on frappedesk.com). |

**Effort:** M (~3-5 days)
**Risk level:** Low — uses existing Frappe Helpdesk DocTypes exactly as designed

**Pros:**
- Fully native to Frappe Helpdesk — no custom DocTypes needed beyond existing `Haravan Account Link`
- Agents see HD Customer in standard ticket filters, reports, SLA rules
- Contact↔HD Customer linking is a standard Frappe pattern
- Frontend org picker mirrors Frappe's own pattern (proof: the screenshot)

**Cons:**
- `HD Customer` is a simple DocType (name + domain + image) — limited metadata storage
- Category (`orgcat`) from Haravan has no native HD Customer field (need custom field or store in link doc)

**Best when:** You want maximum compatibility with Frappe Helpdesk out-of-the-box features.

---

### Option B: Dual-Layer (HD Customer + Company)

**Approach:** Create both HD Customer (for Helpdesk) AND Company (for Frappe CRM/accounting). HD Customer references Company via custom Link field. Enrichment writes to both layers.

**Philosophy:** Maximum data richness, future-proof for ERPNext

**What changes:**

| Component | Change |
|-----------|--------|
| `enrich_frappe_data()` | Keep Company creation. ADD HD Customer creation. Link HD Customer → Company. |
| HD Customer | Add custom fields: `haravan_orgid`, `haravan_orgcat`, `company` (Link to Company). |
| Contact | Link to both Company and HD Customer. |
| API + Frontend | Same as Option A (org picker based on HD Customer). |

**Effort:** L (~5-8 days)
**Risk level:** Medium — dual sync adds complexity and potential for inconsistency

**Pros:**
- Richest data model — supports future ERPNext / CRM integration
- Company doc provides `default_currency`, accounting chart of accounts, etc.
- HD Customer provides Helpdesk-native ticket association

**Cons:**
- Two entities to keep in sync for each Haravan org
- Company DocType requires mandatory fields (`abbr`, `default_currency`) that may conflict
- Over-engineering for a Helpdesk-only use case

**Best when:** You plan to use Frappe ERPNext (invoicing, accounting) alongside Helpdesk in the future.

---

### Option C: Custom "Haravan Organization" DocType

**Approach:** Create a new custom DocType `Haravan Organization` that stores all Haravan-specific fields. HD Ticket gets a custom Link field pointing to it instead of using HD Customer.

**Philosophy:** Full control, Haravan-specific

**What changes:**

| Component | Change |
|-----------|--------|
| New DocType: `Haravan Organization` | Fields: `orgid`, `orgname`, `orgcat`, `domain`, `image`, `locale`. |
| HD Ticket | Add custom field `haravan_organization` (Link to Haravan Organization). |
| Contact | Link via Dynamic Link to Haravan Organization. |
| Enrichment | Write to Haravan Organization instead of Company or HD Customer. |
| API + Frontend | Org picker uses Haravan Organization list. |

**Effort:** L (~5-8 days)
**Risk level:** High — breaks compatibility with Helpdesk's native customer model

**Pros:**
- Can store all Haravan-specific metadata (orgcat, orgid, locale, etc.)
- Clean separation from Frappe's standard entities
- Full control over schema evolution

**Cons:**
- HD Ticket's native `customer` field (HD Customer) is bypassed — agents lose standard filtering/SLA
- Helpdesk reporting won't recognize the custom org field
- Requires modifying Helpdesk's Vue frontend (not just injecting JS)
- More DocTypes to maintain, more migration complexity

**Best when:** Haravan org metadata is business-critical and HD Customer is insufficient (unlikely for helpdesk-only use case).

## 🎯 Recommendation

### Choose Option A: HD Customer Native Integration

#### Why Option A:

1. **Platform-native** — HD Customer is exactly the entity Frappe Helpdesk was designed to link tickets to. The screenshot from frappedesk.com proves Frappe itself uses this pattern (Site Name → HD Customer).

2. **Minimal friction** — No new DocTypes needed. The `Haravan Account Link` already stores the mapping data. We just need to sync to HD Customer instead of Company.

3. **Fastest to deliver** — 3-5 days to complete. All components use existing Frappe patterns (Link fields, whitelisted APIs, web_include_js).

#### Why NOT the other options:

- **Option B (Dual-Layer):** Over-engineering. Company docs are for accounting/ERP, not helpdesk. Creating both adds sync complexity with no helpdesk benefit. Can always add Company sync later if ERPNext is deployed.

- **Option C (Custom DocType):** Breaks Helpdesk's native customer model. Agents lose standard filters, SLA rules, and reporting. The cure is worse than the disease.

#### Quick-Win Path:

Start with the backend changes (Steps 1-3 below), which will immediately fix the data model. The frontend org picker (Step 4) can follow in a second PR.

### Step-by-Step Scope

#### Step 1: Fix Data Model — Replace Company with HD Customer

**Files:** `login_with_haravan/engines/sync_helpdesk.py` (new), `login_with_haravan/oauth.py`

- Create new engine `sync_helpdesk.py` with:
  - `upsert_hd_customer(orgid, orgname, orgcat)` — create/update HD Customer doc
  - `ensure_contact_linked_to_customer(contact_name, hd_customer_name)` — add Dynamic Link
  - `update_user_profile(user, middle_name, locale)` — update User doc fields
- Replace `enrich_frappe_data()` in `oauth.py` with call to new engine
- Remove Company creation logic entirely
- Remove debug file-dump code (lines 94-100)

#### Step 2: Enhance Haravan Account Link DocType

**Files:** DocType JSON schema

- Add field `hd_customer` (Link to HD Customer) — tracks which HD Customer was created
- This enables querying: "for user X, what HD Customers exist?" via Haravan Account Link

#### Step 3: Whitelisted API for Org Listing

**Files:** `login_with_haravan/oauth.py` or new `login_with_haravan/api.py`

```python
@frappe.whitelist()
def get_user_haravan_orgs():
    """Return list of HD Customers linked to current user via Haravan Account Link."""
    user = frappe.session.user
    links = frappe.get_all(
        "Haravan Account Link",
        filters={"user": user},
        fields=["haravan_orgid", "haravan_orgname", "hd_customer"]
    )
    return [
        {"orgid": l.haravan_orgid, "orgname": l.haravan_orgname, "customer": l.hd_customer}
        for l in links if l.hd_customer
    ]
```

#### Step 4: Frontend Org Picker (Portal)

**Files:** `login_with_haravan/public/js/haravan_org_selector.js`

- On helpdesk `/my-tickets/new` page:
  - Call `get_user_haravan_orgs`
  - If 0 orgs: hide picker, ticket has no customer
  - If 1 org: auto-set customer field (no UI needed)
  - If 2+ orgs: render dropdown like frappedesk.com's "Site Name"
- On ticket submit: ensure `customer` field is set to selected HD Customer

#### Step 5: Doc Events Hook (Fallback)

**Files:** `login_with_haravan/hooks.py`

```python
doc_events = {
    "HD Ticket": {
        "before_insert": "login_with_haravan.engines.sync_helpdesk.auto_set_customer"
    }
}
```

- If `customer` is empty and user has exactly 1 linked HD Customer → auto-set
- If user has multiple → leave empty (portal JS should have set it)

#### Step 6: Tests

- Unit tests for `sync_helpdesk.py` (mock frappe.get_doc/new_doc)
- Test `get_user_haravan_orgs` returns correct structure
- Test `auto_set_customer` fallback logic

### Key Constraints

- Do NOT modify Frappe Helpdesk core or its Vue frontend
- Use only `web_include_js` for frontend injection
- All operations must use `ignore_permissions=True` (OAuth callback runs as system)
- HD Customer `customer_name` must be unique — use org_name as-is, handle duplicates with orgid suffix

### Open Questions for Implementation

1. **HD Customer naming:** Should we use `orgname` directly (e.g., "Minh Hải Store") or prefix with orgid (e.g., "HRV-12345 Minh Hải Store")? Recommendation: use orgname, append orgid only on collision.
2. **Existing Company docs:** Should we migrate the Company docs already created by `enrich_frappe_data()` to HD Customer? Recommendation: yes, via a one-time migration patch.
3. **orgcat field:** HD Customer has no `category` field. Store in Haravan Account Link only, or add custom field to HD Customer? Recommendation: custom field on HD Customer.
